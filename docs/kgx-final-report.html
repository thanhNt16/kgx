Retrieval Pipeline Scale Rerun

The 10k-node benchmark was rerun after preserving the 220-note base corpus inside the generated large vault and adding final-stage title deduplication to prevent repeated generated notes from consuming the top five. The corrected benchmark now passes every retrieval gate.

Executive Summary

The previous 10k-node run reported a recall collapse from 0.7333 to 0.2000. Investigation found that the scale generator had not preserved the base benchmark corpus, so many gold patterns such as backpressure, billing_ledger, exactly-once, and partition spec were absent from the generated vault. Those questions were impossible to answer.

After using a large vault generated with --base-vault /tmp/kgx-bench-vault and deduplicating repeated generated titles before applying the result limit, the rerun recovered to recall@5 0.8667, MRR 0.8019, and p95 latency 115.14 ms. This confirms that the largest failures were benchmark-data invalidity and duplicate-result saturation, not an inherent retrieval collapse.

What Changed

The checked-in benchmark scripts already include the required fixes: bench/gen_large_corpus.py supports --base-vault, and bench/bench.py rejects unreachable gold entries before scoring.

Large Corpus Generation

  • Copies base vault notes/ and raw/ into the generated large vault.
  • Subtracts copied base notes from the synthetic target.
  • Records synthetic_nodes and base_nodes in manifest.json.

Benchmark Reachability Guard

  • Loads note bodies before running retrieval.
  • Checks each gold item by note id or expected pattern.
  • Exits with a clear error when a gold item cannot exist in the target vault.

Final Result Deduplication

  • Deduplicates final search results by normalized title before applying --limit.
  • Stores notes.title in the brain index so dedup does not parse note bodies during search.
  • Collapses generated headings such as DL-40918 (Sprint 309): ... to their semantic title.
  • Prevents repeated synthetic notes from consuming multiple top-five slots.

Recommended Next Fix

The next implementation should focus on increasing candidate recall before final ranking. The comparable mock-reranker probe degraded vocab-mismatch recall, so the release profile keeps reranking off.

1. Make Candidate Depth Configurable

  • Add candidate_limit and ppr_limit to SearchOpts.
  • Use 200-500 internal candidates for deeper diagnostic runs.
  • Keep returned --limit at 5 so benchmark semantics do not change.

2. Add Rank-Depth Diagnostics

  • Record whether each gold item appears in BM25, LIKE, sparse, dense, PPR, and fused candidate pools.
  • Report first relevant rank beyond top five.
  • Separate "not generated" from "generated but misranked".

3. Improve Vocab-Mismatch Retrieval

  • Test real SPLADE on the preserved-base 10k vault when CPU time allows.
  • Consider synonym/entity expansion for terms like "page" to "PagerDuty" and "stream processing" to "Flink".
  • Retune RRF weights after candidate-depth instrumentation exists.

4. Keep Latency Profiles Separate

  • Interactive profile: keep p95 under 200 ms.
  • High-recall profile: allow deeper candidates and optional reranker.
  • Do not hide expensive ONNX stages behind a single default gate.

Release status: the corrected 10k retrieval gate is green. The remaining work is quality improvement beyond the current acceptance floor, especially semantic bridging for the three remaining vocab-mismatch misses.
